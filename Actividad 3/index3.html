Introducción

El Lenguaje Unificado de Modelado (UML) es un lenguaje estándar para especificar, visualizar, construir y documentar los artefactos de un sistema. UML proporciona un conjunto de notaciones que permiten a los desarrolladores y arquitectos de software describir de manera precisa los componentes, relaciones y comportamientos de un sistema.

¿Qué es UML?

UML es un lenguaje utilizado para visualizar, especificar, construir y documentar los componentes de un sistema de software. Es especialmente útil en el desarrollo de software orientado a objetos, pero también se puede aplicar en otros campos como la ingeniería de sistemas y la gestión de procesos empresariales.

Principales características

  • Lenguaje modelado: UML no es un lenguaje de programación, sino un lenguaje para crear diagramas que describen sistemas de software. Estos diagramas ayudan a entender la estructura y el comportamiento de un sistema.
  • Notación estándar: UML proporciona una notación estándar aceptada mundialmente que permite a diferentes equipos trabajar de forma coherente.
  • Flexible: Puede aplicarse en diferentes metodologías de desarrollo de software, ya sea en enfoques ágiles, en cascada u otros.
  • Modelado de sistemas orientados a objetos: UML está diseñado para representar sistemas orientados a objetos, aunque también puede usarse en otros tipos de sistemas.

Diagramas UML

Se dividen en dos secciones:

  • Diagramas Estructurales
    • Diagrama de clases: Muestra las clases de un sistema, sus atributos, métodos y las relaciones entre ellas.
    • Diagrama de objetos: Representa instancias específicas de clases en un momento particular.
    • Diagrama de componentes: Describe los componentes físicos del sistema y las interacciones entre ellos.
    • Diagrama de despliegue: Muestra la arquitectura de hardware del sistema y cómo los componentes de software se distribuyen.
    • Diagrama de paquetes: Organiza elementos de un sistema en grupos para mostrar su estructura modular.
  • Diagramas Comportamentales
    • Diagrama de casos de uso: Muestra las interacciones entre los actores externos y el sistema.
    • Diagrama de secuencia: Representa la interacción entre objetos a lo largo del tiempo.
    • Diagrama de actividad: Representa flujos de trabajo o procesos dentro del sistema.

Patrones de Diseño

Los patrones de diseño son soluciones generales y reutilizables a problemas comunes en el diseño de software.

  • Singleton: Asegura que una clase tenga una única instancia y proporciona un punto de acceso global a ella.
  • Factory Method: Define una interfaz para crear un objeto, pero permite a las subclases decidir qué clase instanciar.
  • Observer: Permite que un objeto notifique a otros objetos sobre cambios en su estado.
  • Decorator: Permite añadir comportamientos o responsabilidades a un objeto de manera dinámica.

Glosario de Terminología de UML

  • Actor: Entidad externa al sistema, que puede ser un usuario, un sistema o una organización, que interactua con el sistema para alcanzar un objetivo.
  • Asociación: Relación entre dos o más clases que indica que los objetos de una clase están conectados a los objetos de otra. Puede ser unidireccional o bidireccional.
  • Atributo: Propiedad o característica de una clase que describe los datos que pueden ser almacenados por las instancias (objetos) de esa clase.
  • Clase: Una plantilla para crear objetos, que define un conjunto de atributos y métodos.
  • Composición: Plantilla que define los atributos y comportamientos de los objetos que pertenecen a ella. Es el núcleo de los diagramas de clases.
  • Componente: Tipo de relación entre clases donde una clase está compuesta de otras. Si la clase contenedora se destruye, las clases componentes también se destruyen (relación fuerte).
  • Dependencia: Relación entre dos elementos en la que un cambio en un elemento (dependiente) podría afectar al otro (del que depende).
  • Encapsulamiento: Principio que consiste en ocultar los detalles internos de una clase, de manera que los datos solo puedan ser manipulados a través de sus métodos públicos.
  • Generalización: Relación de herencia entre clases, donde una clase hija (subclase) hereda atributos y métodos de una clase padre (superclase).
  • Herencia: Mecanismo por el cual una clase (subclase) hereda propiedades y comportamientos de otra clase (superclase). Es representada como una flecha con línea sólida y un triángulo hueco en los diagramas de clases.
  • Instancia: Especificación de un objeto real que pertenece a una clase. En UML, una instancia se muestra en diagramas de objetos.
  • interfaz Conjunto de operaciones que una clase o componente debe implementar. Define un contrato que las clases deben cumplir, pero no especifica cómo hacerlo.
  • Mensaje: Comunicación entre objetos en un diagrama de interacción (secuencia o comunicación) que indica la invocación de un método o el paso de información.
  • Método: Función o procedimiento definido en una clase que describe un comportamiento de sus objetos.
  • Nodo: Elemento físico en un diagrama de despliegue que representa un dispositivo o máquina donde se ejecuta o almacena el software.
  • Objeto: Una instancia de una clase, que tiene un estado y un comportamiento.
  • Paquete: Agrupación lógica de clases u otros elementos en un diagrama de paquetes. Sirve para organizar y estructurar grandes sistemas.
  • Polimorfismo: Capacidad de diferentes clases para responder de manera distinta al mismo mensaje o método. Facilita la flexibilidad del código.
  • Relación: Función o comportamiento específico que un objeto o actor adopta en una relación o interacción en un diagrama de casos de uso o diagramas de interacción.
  • Secuencia: Orden en el que ocurren las interacciones o los mensajes entre objetos. Se representa en los diagramas de secuencia.
  • Estado: Situación o condición actual de un objeto en un diagrama de estados. Cambia como resultado de eventos o transiciones.
  • Transición: Cambio de un objeto o sistema de un estado a otro en respuesta a un evento, representado en un diagrama de estados.
  • Caso de uso: Descripción de cómo un actor externo interactúa con el sistema para lograr un objetivo. Modela los requisitos funcionales del sistema.
  • Visibilidad: Propiedad que define qué elementos pueden acceder a los atributos o métodos de una clase. Los niveles comunes son público, privado, y protegido.
